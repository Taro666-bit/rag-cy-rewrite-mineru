1. 这个项目是干什么的？

这是一个RAG 智能问答系统——用 AI 自动回答关于公司年报的专业问题。

把它想象成一个超级聪明的「年报阅读机器人」：你给它一堆 PDF 年报，然后问它问题（比如「中芯国际 2024 年营收是多少？」），它就能自动从 PDF 里找到答案并回答你。

举个具体例子

假设你上传了 9 份关于中芯国际的 PDF 研报（券商报告、公司年报等），然后问：

你的问题系统做什么最终回答
「中芯国际在晶圆制造行业中的地位如何？」 从 PDF 中搜索相关内容 → LLM 总结 中芯国际是中国大陆最大的晶圆代工企业，全球排名第五…
「中芯国际 2024 年营收是多少？」 精准定位财务数据页面 → 提取数值 578.9（亿元）

支持的问题类型

类型示例答案格式
string 文本「请总结中芯国际的主营业务」一段完整文字
number 数值「中芯国际 2024 年营收是多少？」578.9
name 人名「中芯国际的 CEO 是谁？」「张三」
names 名单「中芯国际有哪些新任高管？」['张三', '李四']
boolean 布尔「中芯国际是否宣布了分红政策变更？」true / false
比较类「A 公司和 B 公司哪家营收更高？」「A 公司」

2. 核心概念：什么是 RAG？

RAG = Retrieval-Augmented Generation（检索增强生成）

一句话理解：让 LLM（大模型）在回答之前，先「翻书」找到相关内容，再基于找到的内容作答。这样 LLM 就不会瞎编（幻觉）。

为什么不直接问 ChatGPT？

方式优点缺点
直接问 ChatGPT 简单方便 可能编造数据、不知道你公司特定信息、有知识截止日期
用 RAG 系统 答案基于你提供的文档，可追溯来源 需要额外的检索步骤

6. 第二步：文本规整 → Markdown

src/parsed_reports_merging.py 中的 PageTextPreparation 类负责把结构化 JSON 转换成干净的 Markdown 文本。

它做了什么？

  1. 过滤不需要的内容：移除页脚（page_footer）和图片（picture）
  2. 格式化排版：
    • page_header → # 标题（一级标题）
    • section_header → ## 标题（二级标题）
    • paragraph → ### 段落（三级标题）
    • list_item → - 列表项
  3. 表格处理：把表格和它的标题、脚注合并在一起
  4. 文本清洗：修正 OCR 识别错误。比如把 /zero 替换成 0，把 /percent 替换成 %
想象一下 OCR 扫描时把「100%」错误识别成了「100/percent」——文本清洗就是用正则表达式把这些 OCR 错误修正回来。

转换示例

输入（JSON 块）：
{"type": "section_header", "text": "主营业务分析"}
{"type": "paragraph", "text": "公司2024年实现营业收入578.9亿元"}
{"type": "table", "table_id": 0}

↓ PageTextPreparation 处理 ↓

7. 第三步：文本分块（Chunking）

src/text_splitter.py 中的 TextSplitter 类负责把长文本切成小块。

为什么要分块？因为如果把整本 200 页的年报一次性丢给 LLM，它会「看不过来」（token 限制），检索精度也会很低。 切成小段后，每段对应一个向量，检索时可以精准定位到最相关的几个段落。

当前使用的分块策略（MinerU 路径）

split_markdown_file() 方法：

  • 按行分割：每 30 行切成一个 chunk
  • 滑动重叠：相邻 chunk 之间重叠 5 行（防止关键内容恰好落在边界上）

200行"] --> B["Chunk 1
第1-30行"] A --> C["Chunk 2
第25-55行
（重叠5行）"] A --> D["Chunk 3
第50-80行"] A --> E["..."] B --> F["🗄️ 存入 chunked_reports/*.json"] C --> F D --> F E --> F
MinerU 路径的分块没有页码信息（chunk 只有 lines 字段，没有 page 字段）。 这导致后面的「页码引用」和「父文档检索」功能无法正常工作。

      { "lines": [50, 80],  "text": "...（下一个块）..." }

  }
}

8. 第四步：向量化 & 建库

src/ingestion.py 负责两个关键任务：创建向量库和 BM25 索引。

8.1 什么是向量嵌入（Embedding）？

向量就是把一段文字变成一串数字。比如「营收增长」可能变成 [0.23, -0.45, 0.78, ...]（1536 个数字）。 两段语义相近的文字，它们的向量在空间中距离更近。这就是「语义搜索」的基础。

8.2 两个核心类

类名索引方式搜索原理适用场景
VectorDBIngestor FAISS 向量索引 语义相似度（余弦距离） 理解含义的搜索，如「公司经营情况」
BM25Ingestor BM25 稀疏索引 关键词匹配（TF-IDF 变体） 精确关键词搜索，如「营业收入 578.9亿」

8.3 VectorDBIngestor 的工作流程

sequenceDiagram participant Chunks as 分块JSON participant VDB as VectorDBIngestor participant DashScope as DashScope Embedding API participant FAISS as FAISS 向量库 Chunks->>VDB: 读取所有 chunked_reports/*.json VDB->>VDB: 提取每个 chunk 的 text 字段 VDB->>VDB: 截断超长文本（max 2048字符） VDB->>DashScope: 调用 text-embedding-v1
(每批最多25条) DashScope-->>VDB: 返回 1536 维向量 VDB->>FAISS: 用 IndexFlatIP 创建向量索引 FAISS-->>VDB: 保存为 *.faiss 文件

8.4 关键代码逻辑

# src/ingestion.py - VectorDBIngestor 核心逻辑

def _process_report(self, report: dict):
    # 1. 取出所有文本块
    text_chunks = [chunk['text'] for chunk in report['content']['chunks']]

    # 2. 截断超长文本（最大 2048 字符）
    text_chunks = [t[:2048] for t in text_chunks if len(t) > 0]

    # 3. 批量获取嵌入向量（每批 25 条）
    embeddings = self._get_embeddings(text_chunks)

    # 4. 创建 FAISS 向量索引（内积 = 余弦相似度）

    index.add(embeddings_array)

    return index

9.2 检索器详解（src/retrieval.py）

项目有 3 种检索器，形成递进关系：

检索器机制什么时候用
VectorRetriever 把问题转成向量，在 FAISS 中找最相近的 chunk 基础配置、单公司问题
BM25Retriever 把问题分词，用 BM25 算法匹配关键词 需要精确关键词匹配时
HybridRetriever 向量检索 top-28 → LLM 重排 → 取 top-6 启用 llm_reranking 时

    def send_message(self, system_content, human_content, ...):
        # 不管底层是哪个提供商，统一调用 send_message
        return self.processor.send_message(...)

Number 类型的严格约束（重点）

数值类问题最容易出错，所以在 AnswerWithRAGContextNumberPrompt 中嵌入了7 条严格规则：

  1. 明确问题中指标的精确定义
  2. 检查上下文中的所有可能指标，关注实际含义
  3. 仅当上下文指标含义与目标指标完全一致时才接受
  4. 拒绝的情况：范围不同、概念相似但不完全等价、需要计算推导、聚合不匹配
  5. 不允许猜测：有任何疑问就返回 N/A
  6. 自动处理格式：百分比、负数括号、千分位、币种不匹配
  7. 不可由其他指标计算/推导 → 返回 N/A

数值格式转换举例

原文中出现的值正确答案规则
58,3%58.3逗号→小数点，移除%
(2,124,837) CHF-2124837括号=负数，移除分隔符
4970,5（千美元）4970500千为单位→乘1000
780000 USD（问题要 EUR）N/A币种不匹配

四层约束机制

每个 Prompt 的输出通过四层约束确保质量：

  1. Pydantic Schema：用 Python 类型系统精确定义输出字段
  2. 类型 + Literal 联合：约束 LLM 只能返回特定类型值
  3. Few-shot Example：用示例教 LLM 模仿标准输出格式
  4. Instruction 内嵌规则：在系统提示词中写明禁止推算、禁止猜测

15. 已知问题 & 改进方向

LLMReranker 的 dashscope 分支有 Bug：LLM 的评分结果被硬编码为 0.0，实际没有用上。重排功能退化为纯向量距离排序。 详见 src/reranking.py 第 88-94 行和第 137-142 行。
MinerU 路径缺少页码信息：当前使用的分块方式（按行分割）不记录页码，导致两个功能无效：
  • 父文档检索（parent_document_retrieval）：需要从 chunk 找到它属于哪一页，但 MinerU 路径下 chunks 没有 page 字段
  • 页码校验（_validate_page_references）：依赖 result['page']，MinerU 下该值恒为 0

改进建议

  1. 修复 LLMReranker：解析 dashscope 返回的 JSON 字符串，提取真实的 relevance_score
  2. 换用 DashScope TextReRank：阿里云有专用重排 API（gte-rerank），比用聊天模型打分更合理
  3. 接线 JinaReranker：代码已写好（src/reranking.py 第 10-35 行），只需在 HybridRetriever 中替换引用
  4. 补充 MinerU 路径的页码：如果 full.md 中有分页标记，可在分块时解析并记录
  5. 改用 Docling 路径：如果有 GPU，回到 Docling 路径可以获得完整的页码和结构化信息
