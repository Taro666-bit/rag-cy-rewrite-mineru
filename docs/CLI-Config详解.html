一、Click 是什么？为什么能敲命令？

1.1 核心三件套

整个命令行系统由 Click 库提供。三个核心装饰器构成了命令体系：

装饰器作用比喻
@click.group() 创建一个"命令组"，相当于遥控器底座 🎮 遥控器外壳
@cli.command() 把函数注册为子命令，相当于遥控器上的按钮 🔘 遥控器按钮
@click.option(...) 给命令添加可选参数 🎚️ 按钮旁的调节旋钮

1.2 从代码到命令行的映射

import click

@click.group()                          # ① 造遥控器
def cli():
    pass

@cli.command()                          # ② 装按钮：download-models
def download_models():
    """按下去就执行这个函数"""
    click.echo("Downloading...")

@cli.command()                          # ② 装按钮：parse-pdfs
def parse_pdfs():
    """按下去就执行这个函数"""
    click.echo("Parsing...")

if __name__ == '__main__':
    cli()                                 # ③ 按下启动键，等待用户输入

命名规则：下划线 → 短横线

Click 会自动把 Python 函数名里的下划线 _ 转成命令行里的短横线 -：

Python 函数名→命令行命令
download_models→python main.py download-models
parse_pdfs→python main.py parse-pdfs
process_questions→python main.py process-questions

1.3 @click.option 详解

基本形态：数值/字符串参数

@click.option('--chunk-size', default=2, help='...')
def parse_pdfs(chunk_size):   # 参数自动传入
    ...
命令行写法函数收到
（不写）chunk_size = 2
--chunk-size 5chunk_size = 5
--chunk-size 100chunk_size = 100

特殊形态：布尔开关 --xxx/--yyy

@click.option('--parallel/--sequential', default=True)
def parse_pdfs(parallel):    # True 或 False
    ...
命令行写法函数收到含义
--parallelparallel = True走并行模式
--sequentialparallel = False走串行模式
（不写）parallel = True（默认值）走并行模式
关键规则：--xxx/--yyy 中，第一个名字对应 True，第二个名字对应 False。 两者互斥——不能同时写 --parallel --sequential。
就像电灯开关：--parallel 是「开」，--sequential 是「关」。

枚举限制：click.Choice

@click.option('--config', type=click.Choice(['base', 'pdr', 'max']), default='base')
def process_questions(config):
    ...
命令行写法结果
--config base✅ 通过，config = "base"
--config max✅ 通过，config = "max"
--config abc❌ 报错！不在可选列表中

1.4 装饰器的执行顺序

装饰器从下往上执行（离函数近的最先）。下面三行等价于：

@cli.command()                              # ③ 最后：注册为 CLI 子命令
@click.option('--parallel/--sequential')   # ② 然后：添加选项
@click.option('--chunk-size', default=2)         # ① 最先：添加选项
def parse_pdfs(parallel, chunk_size):

最终效果：parse_pdfs 变成 cli 的子命令 parse-pdfs，接受 --parallel 和 --chunk-size 两个可选参数。

二、全部 5 条命令详解

所有命令都以 python main.py 开头运行。下面是每条命令的完整说明。

① download-models — 一次性准备工作

python main.py download-models
什么时候跑只跑一次，环境初始化
干什么下载 Docling（PDF 解析引擎）需要的 AI 模型文件
输入无
输出模型文件存入本地缓存
代码路径main.py → Pipeline.download_docling_models() → PDFParser.parse_and_export()

② parse-pdfs — PDF → 结构化 JSON

# 全部默认（并行、每worker 2个PDF、10个worker）
python main.py parse-pdfs

# 串行模式（调试用）
python main.py parse-pdfs --sequential

# 自定义参数
python main.py parse-pdfs --parallel --chunk-size 5 --max-workers 4
参数类型默认说明
--parallel / --sequential布尔开关True并行（快）/ 串行（调试）
--chunk-size整数2每个 worker 一次处理几个 PDF
--max-workers整数10最多几个进程同时干活

执行流程

pdf_reports/*.pdf │ ▼ Docling 引擎解析 │ ▼ debug_data/01_parsed_reports/*.json ← 结构化 JSON（文本、表格、页码） debug_data/01_parsed_reports_debug/*.json ← 调试版本

③ serialize-tables — 表格优化（可选）

python main.py serialize-tables --max-workers 10
干什么把 PDF 里的表格转成 LLM 容易读的纯文本描述
什么时候跑parse-pdfs 之后、process-reports 之前。PDF 表格不多可跳过
代码路径main.py → Pipeline.serialize_tables() → TableSerializer

④ process-reports — 分块 + 向量化 + 建 FAISS 索引

# 基础版：只分块，不用表格序列化
python main.py process-reports --config no_ser_tab

# 增强版：分块时带上序列化后的表格
python main.py process-reports --config ser_tab
参数类型可选值
--config枚举no_ser_tab（默认）、ser_tab

执行流程

debug_data/03_reports_markdown/*.md │ ▼ [step1] TextSplitter.split_markdown_reports() │ 按 30 行一块切，重叠 5 行 │ ▼ databases/chunked_reports/*.json ← 切好的文本块 │ ▼ [step2] VectorDBIngestor.process_reports() │ 每个 chunk → DashScope 生成向量 → FAISS 建索引 │ ▼ databases/vector_dbs/*.faiss ← 向量索引（每公司一个）
注意：如果 databases/vector_dbs/ 下已经有 *.faiss 文件，不需要重跑这个命令！

⑤ process-questions 🔥 核心命令 — 检索 + LLM 回答

# 基础版：快、便宜、GPT-4o-mini
python main.py process-questions --config base

# 增强版：父文档检索、GPT-4o
python main.py process-questions --config pdr

# 旗舰版：父文档检索 + LLM重排、Qwen-Turbo ⭐推荐
python main.py process-questions --config max
参数类型可选值
--config枚举base、pdr、max 等

三、Config 系统详解

3.1 两层配置

第一层：PipelineConfig — 管「文件放哪里」的路径配置

class PipelineConfig:
    def __init__(self, root_path, serialized=False, config_suffix=""):
        self.subset_path      = root_path / "subset.csv"
        self.questions_file   = root_path / "questions.json"
        self.answers_file     = root_path / f"answers{config_suffix}.json"
        self.vector_db_dir    = databases_path / "vector_dbs"
        self.documents_dir    = databases_path / "chunked_reports"
        self.bm25_db_path     = databases_path / "bm25_dbs"
        ...

📌 比喻：就像你决定「今天所有照片存到 旅行/日本/ 文件夹」。只管路径，不管怎么干活。

第二层：RunConfig — 管「程序怎么跑」的行为配置

class RunConfig:
    use_serialized_tables: bool        = False   # 是否用表格序列化
    parent_document_retrieval: bool    = False   # 是否返回整页内容
    llm_reranking: bool                = False   # 是否用 LLM 重排
    llm_reranking_sample_size: int     = 30      # 重排时粗筛多少条
    top_n_retrieval: int               = 10      # 最终保留多少条上下文
    parallel_requests: int             = 1       # 同时处理几个问题
    answering_model: str               = "qwen-turbo"  # 用哪个模型
    api_provider: str                  = "dashscope"   # 用哪个 API
    config_suffix: str                 = ""        # 输出文件名后缀

📌 比喻：就像相机的「人像模式/夜景模式/运动模式」，选不同模式会调整快门、ISO、对焦方式。

3.2 参数逐个解释

参数名类型作用max 的值
parent_document_retrieval bool 检索时是否返回匹配 整页 内容（而非仅匹配的小段落）。
就像查书时不仅看那一句话，还把整页都读了。		 ✅ True
llm_reranking bool 检索结果是否再用 LLM 重新打分排序，把最相关的排最前面。
就像搜索引擎的「按相关度排序」。		 ✅ True
llm_reranking_sample_size int 重排序时先粗筛多少条候选（从里面精选）。
数字越大越准但越慢。		 30
top_n_retrieval int 最终保留多少条相关结果喂给 LLM。
太多会超 token 限制，太少可能信息不足。		 10
parallel_requests int 同时处理几个问题。受 API 限流制约。
Qwen 限流严格所以只设 4，GPT 可设 10~20。		 4
answering_model str 用于生成最终答案的模型名称。 "qwen-turbo"
api_provider str 调用哪个 API 服务商（dashscope / openai）。 "dashscope"
config_suffix str 输出文件名的后缀。如 "_qwen_turbo" → answers_qwen_turbo.json "_qwen_turbo"
use_serialized_tables bool 报告分块时是否插入序列化后的表格内容。 ❌ False

3.3 Config 的流转链路

你敲的命令 代码里的动作 ══════════ ══════════ --config max ──────────→ configs["max"] ──→ max_config (RunConfig 对象) │ ▼ Pipeline(root_path, run_config=max_config) │ ├── self.run_config = max_config │ ├── PipelineConfig(config_suffix="_qwen_turbo") │ ├── answers_file → answers_qwen_turbo.json │ └── vector_db_dir → databases/vector_dbs/ │ ▼ QuestionsProcessor( parent_document_retrieval=True, ← 来自 config llm_reranking=True, ← 来自 config parallel_requests=4, ← 来自 config answering_model="qwen-turbo", ← 来自 config top_n_retrieval=10, ← 来自 config ) │ ▼ 实际干活：检索 → 重排 → LLM → 生成答案

3.4 config 决定检索器的分支逻辑

# 在 questions_processing.py 的 get_answer_for_company() 中：
if self.llm_reranking:
    retriever = HybridRetriever(...)   # 向量检索 + LLM重排（max）
else:
    retriever = VectorRetriever(...)   # 纯向量检索（base/pdr）

HybridRetriever 的内部流程：

  1. 向量检索粗筛 30 条（llm_reranking_sample_size）
  2. LLM 对这 30 条重新打分排序
  3. 取前 10 条（top_n_retrieval）作为最终上下文

3.5 如何自定义配置

假设你想做一个「超快版」：不重排、不检索父文档，但并行开到 20：

# 在 src/pipeline.py 里加上：
fast_config = RunConfig(
    parent_document_retrieval=False,
    llm_reranking=False,
    parallel_requests=20,
    answering_model="qwen-turbo",
    config_suffix="_fast"
)

# 注册到字典里：
configs = {
    "base": base_config,
    "pdr":  pdr_config,
    "max":  max_config,
    "fast": fast_config,   # ← 新增
}

然后就可以：

python main.py process-questions --config fast

3.6 Streamlit 界面也走 Config

app_streamlit.py 固定使用 max_config：

from src.pipeline import Pipeline, max_config

pipeline = Pipeline(root_path, run_config=max_config)

所以你在网页界面输入问题，走的就是 max 配置的完整路径。

五、常见问题 & 速查表

我只想回答问题，前面的步骤都必须跑吗？

不需要！ 如果 databases/vector_dbs/ 下已经有 *.faiss 文件了，直接：

python main.py process-questions --config max

①②③④ 只在第一次或者换了新的 PDF 时才需要跑。

base 和 max 的准确度差距有多大？

base：你去图书馆，管理员随手给你 10 本书 → 自己翻

max：管理员海选 30 本 → 请专家帮你挑最相关的 10 本 → 把关键章节折好角给你

结果质量差距明显。max 每次多调一次 LLM 做重排，所以慢但准。
